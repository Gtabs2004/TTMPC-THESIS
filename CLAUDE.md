## Project Overview

REGANT is a web-based Decision Support System (DSS) for the Tubungan Teachers' Multi-Purpose Cooperative (TTMPC). It combines cooperative operations management (loans, savings, membership) with ML-driven analytics (credit risk scoring, loan demand forecasting, MIGS scoring).

## Commands

### Frontend (inside `TTMPC_THESIS/`)
```bash
npm run dev       # Start dev server at http://localhost:5173
npm run build     # Production build → dist/
npm run preview   # Preview production build
npm run lint      # ESLint
```

### Backend (FastAPI, inside `TTMPC_THESIS/src/server/`)
```bash
uvicorn main:app --reload --host 127.0.0.1 --port 8000
```
Dependencies: `pip install -r requirements.txt` (Python 3.13, use `.venv/`)

### Environment
Frontend requires a `.env` in `TTMPC_THESIS/` with:
- `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`
- `VITE_API_BASE_URL=http://127.0.0.1:8000`
- Resend API key for email notifications

## Architecture

### Role-Based Access Control
The app has 8+ distinct user roles, each with its own portal and route namespace:
- **Member** — self-service (loans, savings, profile)
- **Cashier** — transaction recording (deposits, withdrawals, payments, disbursements)
- **Bookkeeper** — loan evaluation, MIGS scoring, ledger management, POS
- **Manager** — loan approval review
- **Treasurer** — disbursement, accounting, payment reconciliation
- **BOD** (Board of Directors) — member approval, policy setting, secretary functions
- **Non-member/KOICA** — limited loan access for non-members

All 150+ routes are defined in `src/Router.jsx`. Auth context (`src/contex/AuthContext.jsx`) stores role, membership ID, and session state. Note: the context directory is spelled `contex` (not `contexts`).

### Data Flow
- **Database:** Supabase (PostgreSQL) with Row-Level Security enforced per role
- **Auth:** Supabase Auth with `sessionStorage` (not `localStorage`) for multi-tab isolation
- **Frontend → DB:** Direct Supabase client calls (`src/supabaseClient.js`) for most CRUD
- **Frontend → Backend:** FastAPI at port 8000 for ML inference, loan computation, and email

### ML/Analytics Backend (`src/server/`)
Three analytical engines, all served via FastAPI:

1. **Credit Risk (`risk_model.py`)** — Logistic Regression model (`ttmpc_credit_risk_model.pkl`)
   - Endpoint: `POST /score-loan`
   - Key inputs: `LoanAmount`, `Stability_Score`, `Advance_Payment_Count`, `Income_Is_Missing`, `Repayment_Stress_Index`
   - Output: default probability (0–1)

2. **Loan Demand Forecasting (`demand_model.py`)** — SARIMAX models per loan type
   - Endpoint: `GET /demand-forecast`
   - Separate PKL files for consolidated and emergency loans (~7 MB each)
   - Output: 12-month forecast with 80% confidence intervals
   - Displayed in `src/components/LoanDemandForecastCard.jsx`

3. **MIGS Scoring (`migs_engine.py`)** — Pure function, no DB calls
   - 7 scoring criteria totaling 100 points; threshold 50 = MIGS classification
   - Loan multiplier: 5× (MIGS) vs 3× (Non-MIGS)
   - Used by Bookkeeper portal `src/Bookkeeper/Components/MIGS.jsx`

### Loan Lifecycle
Applications flow through roles in sequence:
1. Member applies via `src/LOANFORMS/` → submitted to DB
2. Bookkeeper evaluates & performs MIGS scoring
3. Manager reviews and approves/rejects
4. Treasurer disburses funds
5. Cashier records payments

Loan types: Consolidated, Bonus, Emergency (member), and KOICA (non-member).

### PDF Generation
Loan forms are generated by overlaying data onto existing PDF templates in `src/PDF/`. Uses `pdf-lib` for field stamping and `jsPDF`/`html2canvas` for HTML-to-PDF. See `PDF_EDITING_GUIDE.md` for template field coordinates.

### POS / Grocery Integration
The Cashier and Bookkeeper portals include grocery transaction management. Webhook integration details are in `POS_WEBHOOK_INTEGRATION_NOTES.md`.

### Email Notifications
Sent via Resend API through FastAPI endpoints (`/send-loan-status-email`, `/create-loan-notification`). Templates in `src/server/services/loan_email_templates.py`.

### Database Schema
SQL schema files live in `src/server/*.sql`. Key triggers:
- CBU auto-sync from loan disbursements (`cbu_sync_from_loan_disbursement_include_emergency.sql` is the current version)
- Audit trail for all member/loan modifications

Migration scripts are in `src/server/migration/` — these are one-off data migration utilities, not part of normal app operation.
